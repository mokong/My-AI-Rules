# Changelog 与文档规范（通用模板）

适用于 **应用发版**、**SDK / 框架**、**内部组件库** 等；各项目可删减双语、Emoji 等条目。  
**组件级 API 文档**（如 antd）请在本模板基础上增加「API 命名规则」链接（见贵项目 Wiki）。

**Canonical 源码**：`/Users/apple/Desktop/Project/Doc/ai-generic/docs/`（与本文件同目录）；同步后各工程见 `Docs/Guides/CHANGELOG_AND_DOCUMENTATION_GUIDE.md`。

## 一、作为 Rule 还是 Skill？

| 形式 | 适用场景 |
|------|----------|
| **Skill**（推荐承载全文） | 写 Release Note、整理 CHANGELOG、大版本文档更新时 **按需唤起**；步骤长、示例多，不适合常驻每条对话。 |
| **Rule（.mdc / 短规则）** | 仅保留 **1～3 句**：例如「发版说明须更新 CHANGELOG；格式见 `Docs/Guides/CHANGELOG_AND_DOCUMENTATION_GUIDE.md`」，或 `globs: "**/CHANGELOG*"` 时注入摘要。 |

**结论**：**规范正文用 Skill 或独立本文档；Cursor/Agent 用极短 Rule 指向本文即可。**

## 二、Changelog 基本要求

### 2.1 版本与语言

- **语义化版本**（或项目约定）：`MAJOR.MINOR.PATCH`；**Breaking** 在条目中明确标出。  
- **双语**：若面向中英文用户/开发者，维护 `CHANGELOG.md` + `CHANGELOG.zh-CN.md`（或单文件内中英分段）；单语市场可只保留一种语言。  
- **新增对外 API**（属性、方法、路由、配置项）：在文档或条目中注明 **自哪个版本起可用**。

### 2.2 写什么、不写什么

- **写**：用户或集成方 **能感知** 的行为变化——崩溃修复、交互/流程变更、新能力、废弃、不兼容、性能体感、国际化文案。  
- **不写或降级处理**：纯文档错字、微小样式微调、仅格式化/重命名且无行为变化、内部仅开发者可见的重构——**不写入**主 CHANGELOG，或归入「维护」类一笔带过（团队可自定阈值）。  
- **非产品面向改动**（工程化、CI、仅开发流程）：单独区块标题如 **Infrastructure** / **内部**，或统一记为 **「-」**（与 antd 类似），避免与用户可见变更混淆。

### 2.3 叙述角度

- **面向开发者/用户的影响**，而非实现细节。  
- 描述 **「原先有什么问题 / 现在变成什么样」**，避免「修复了某某内部实现」。

| 不佳 | 更佳 |
|------|------|
| 修复 `ProfileHeader` 内部约束计算顺序 | 修复个人页头部在旋转屏后布局错位的问题 |
| Refactor network layer | （若无行为变化可不写；若有）请求失败时错误提示更准确 |

### 2.4 结构与引用

- **按版本倒序**；同一模块/组件的多条改动 **归并在一起**，子项缩进列表。  
- 尽量附 **PR / Issue 链接**；社区贡献注明 **作者**（如 `@username` 或 PR author）。  
- 依赖 **传递升级**（如底层 UI 库中间版本）：若影响对外行为，在说明中概括 **上游变更要点**。

### 2.5 Emoji（可选，统一团队风格时使用）

| Emoji | 含义 |
|-------|------|
| 🐞 | Bug 修复 |
| 💄 | 样式 / 主题 token |
| 🆕 | 新特性、新 API |
| 🔥 | 重点新特性 |
| 🌐 / 🇺🇸🇨🇳 | 国际化与文案 |
| 📖 📝 | 文档与站点 |
| ✅ | 测试用例增改 |
| 🛎 | 警告、提示文案 |

## 三、文档规范（通用）

### 3.1 与 Changelog 对齐

- 新对外能力：**文档 + CHANGELOG 同步**；注明 **最低支持版本**。  
- **命名**：遵循项目公开的 **API / 路由 / 配置项命名规范**（链接到团队 Wiki）；无则与现有代码风格一致。

### 3.2 建议章节（按产品类型取舍）

- **使用场景**、**快速上手**、**API / 配置说明**、**示例**（短而可运行）。  
- 复杂能力：**详细说明**、边界与限制。  
- **类型定义**：TypeScript / Swift 公开接口 / 头文件声明等，与栈一致。  
- **FAQ**、**迁移指南**（Breaking 时必备）。

### 3.3 维护

- 示例与真实 API **一致**；大改时 **同步所有语言版本**（若双语）。

## 四、与 iOS / 客户端常见结合点

- **发版说明**（App Store / 用户可见）与 **CHANGELOG**（开发者）可同源摘要或分文件。  
- **Feature Flag / 远程配置**：若影响行为，在 CHANGELOG 说明默认与回滚策略（如适用）。
