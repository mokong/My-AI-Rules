# iOS 开发基础规范（跨项目通用）

本文档为 **纯 Markdown**，不依赖某一编辑器的规则格式；适用于任意 Agent / 人工协作。  
**具体工程**的目录、构建命令、路由、埋点、本地化路径等，写在各仓库的 **CLAUDE.md**（或 **AGENTS.md**）中，并注明「通用规范见本文档」。

**Canonical 源码**：本机维护于 **`Doc/ai-generic/docs/`**（例如 `/Users/apple/Desktop/Project/Doc/ai-generic/docs/`）。各工程经 **`sync-generic-ai-from-doc.sh`** 同步后，仓库内路径多为 **`Docs/Guides/IOS_DEVELOPMENT_BASE.md`**。

---

## 一、格式选型说明（为何用 MD 而非 MDC）

| 载体 | 跨编辑器通用性 | 说明 |
|------|----------------|------|
| **`.mdc`（YAML frontmatter + 正文）** | **低** | 主要为 **Cursor Rules** 设计：`globs` / `alwaysApply` 等字段其他工具常忽略或仅当普通文本；换项目需各编辑器各配一份触发方式。 |
| **`CLAUDE.md` / `AGENTS.md` / 普通 `.md`** | **高** | 无工具专有语法；Claude Code、Cursor、Codex、Windsurf 等普遍会读项目根说明；也可复制进 User Rules。 |

**推荐做法**

1. **唯一真相（canonical）**：用 **本类文档（纯 MD）** 或根目录 `AGENTS.md` 存放通用 iOS 规范。  
2. **Cursor 按需增强**：从本文摘短条目写入 `.cursor/rules/*.mdc`，或规则里写一句「详见 `Docs/Guides/IOS_DEVELOPMENT_BASE.md`」。  
3. **各项目专属**：仍用 **CLAUDE.md**（与 Anthropic 工具链习惯一致），开头链回本文。

---

## 二、架构与分层

- **MVVM**：View/ViewController 只做展示与交互；ViewModel 管状态与业务与数据转换；**ViewModel 不持有** UIViewController / SwiftUI `View` / Flutter `Widget` 等视图引用，导航与弹窗通过回调、路由或抽象接口表达；Model 为纯数据（Swift `Codable` / OC 常用模型框架按项目定）；网络与持久化放在 Service 或 Repository。  
- **模块化**：按功能分模块；模块内常见 `Controller` / `View` / `Model` / `Service`（命名可按项目调整）。  
- **模块间**：优先 **协议 + 依赖注入**；跨模块导航用 **路由**（具体 scheme/API 见各项目文档）。  
- **避免**：ViewController 内大段网络与业务、单例滥用、循环依赖。

---

## 三、Swift 约定（类名前缀按团队统一，如 `APP`）

- 类/结构体：**大驼峰**；方法/属性：**小驼峰**；枚举：**大驼峰 + case 小驼峰**。  
- **布局**：若项目约定 Auto Layout 封装库（如 SnapKit），则新代码遵循项目统一写法。  
- **异步**：优先 `async/await`；数据流可用 Combine；UI 相关更新在主线程。  
- **错误**：自定义 `Error`、或 `Result` / `throws`，避免无声失败。  
- **内存**：闭包用 `[weak self]`；`delegate` 用 `weak`；避免循环引用。

---

## 四、Objective-C 约定

- 类名大驼峰 + **团队前缀**；方法/属性小驼峰；文件内常量可用 `k` + 大驼峰。  
- **Block**：`weakSelf` / `strongSelf`，块内使用 `strongSelf` 前判空。  
- **delegate**：`weak`。  
- **网络**：遗留栈与新建抽象并存时，**新代码**优先走项目统一网络层（具体类名见各项目 CLAUDE）。

---

## 五、Swift / OC 混编

- **Bridging Header**：只导入必要 OC 头，控制体积以缩短编译。  
- Swift 需被 OC 调用的 API：**按需 `@objc`**，避免整类 `@objcMembers**。  
- 类型桥接优先 Foundation 友好类型（如 `[String: Any]`）。

---

## 六、可读性与结构（OC/Swift 共通）

- **DRY**：相同逻辑出现两次以上考虑提取；单一职责；参数化变化点。  
- **命名**：**自解释**，见名知义；少无谓缩写；单字母仅用于惯例场景（如循环 `i`）。  
- **组织**：相关代码就近；**一函数一事**；抽象层次一致，避免跳跃。  
- **注释与文档**：注释侧重 **为什么 / 约束 / 陷阱**；**公共 API** 须有清晰文档（Swift 文档注释 / OC 头文件说明），并随代码更新。  
- **魔法数/字符串**：抽为 `static const` / `static NSString * const`（或 Swift `enum`/常量）。  
- **方法长度**：单方法不宜过长（常见上限约 **50 行**，理想 **20～30 行**）；嵌套不宜过深（如 **≤3 层**），多用 **提前返回**。  
- **Category / Extension**：按职责拆分（如 `+UI`、`+Network`），单文件职责清晰、控制方法数量。  
- **错误处理**：不空吞 `@try/@catch`；OC 用 `NSError` 等一致方式；日志 + 用户可读提示 + 必要时上报。

---

## 七、异步与线程

- **UI 必须在主线程**；耗时操作在后台队列。  
- **并发控制**：共享可变状态按项目约定使用锁、串行队列、`actor` 等，避免竞态；避免多余同步。  
- 嵌套 `dispatch_async` 过深时考虑 `dispatch_group` 或拆分方法。

---

## 八、日志

- 区分 Debug / Release：**Release 关闭调试日志**。  
- 分级（Error / Warning / Info 等），带上下文（函数、行号等按项目宏）。  
- **禁止**在日志中输出 token、密码等敏感信息。

---

## 九、禁止事项（摘要）

- VC 内直接复杂网络或核心业务逻辑。  
- 主线程阻塞 I/O 或重计算。  
- 硬编码密钥；忽略证书校验（除非项目明确策略）。  
- 循环引用、过度 `@objc`。

---

## 十、测试与可测试性

- ViewModel、Service 等通过 **协议** 注入依赖，便于 Mock。  
- 单元测试常用 **Given / When / Then**；覆盖成功、空数据、错误、边界等场景。  
- Mock 数据尽量接近真实结构；可用 `#if DEBUG`、Preview 等辅助开发。

---

## 十一、Git 提交（示例约定）

- 类型：`feat` / `fix` / `refactor` / `style` / `docs` / `test` / `chore`。  
- 格式：`<type>: <subject>`，必要时 body + footer（如关联 Issue）。

---

## 十二、性能（通用）

- 减少重复请求；列表 **Cell 复用**；大图/列表图片 **异步加载**（具体库按项目）。  
- **内存**：避免无谓的 **对象创建与复制**（大集合、重复全量 `map` 等按场景优化）；及时释放长生命周期引用，防泄漏。  
- **计算**：避免 **重复计算**；热点路径可缓存；**延迟计算**到确有需要再算。  
- **并行**：可并行任务勿人为串行；注意 **线程安全**（与第七节一致）。  
- 埋点/日志避免阻塞主线程（实现方式见各项目）。

---

## 十三、Agent / 人机协作（可选）

与工具协作时（团队可整段复制到 User Rules 或 Agent 说明）：

1. **语言**：说明与答复优先 **中文**（另有团队约定除外）。  
2. **生成代码**：关键步骤、非显而易见处加 **简短中文注释**（与第六节「注释与文档」一致）。  
3. **输出体量**：单次生成若 **明显超过约 20 行**，宜 **拆函数/拆类型或分步** 交付，便于审查与合并。

---

## 十四、常见坏味道与应对（Fowler 摘要）

| 坏味道 | 处理方向 |
|--------|----------|
| 神秘命名 | 改为描述性名称，使代码自解释 |
| 重复代码 | 提取函数/类/模块；必要时模板方法 |
| 过长函数 | 拆成职责单一的小函数 |
| 过大类/结构体 | 按职责拆类，相关字段与方法归并到新类型 |
| 过长参数列表 | 引入参数对象（如 `UserInfo`） |
| 发散式变化 | 按「变化原因」拆类，单一修改点 |
| 霰弹式修改 | 将分散的相关逻辑收敛到同一模块 |
| 依恋情结 | 方法迁移到更「拥有」其数据的类型 |
| 数据泥团 | 常一起出现的字段抽成小对象（如日期区间） |
| 基本类型偏执 | 用语义小类型替代裸 `String`/`Int`（如电话号码类型） |

---

## 十五、重构过程与范围

- **小步**：每次改动尽量小，保持 **可随时构建、可验证**；宜小步提交。  
- **测试**：重构前后跑自动化测试或关键路径验证；薄弱处可先补最小测试再改。  
- **代码审查**：重要重构走 Review，沉淀经验。  
- **范围边界**：**未在任务中明确要求时**，不做跨模块 **大范围重构**；可先说明问题并给出 **最小可行修改** 再实施。
